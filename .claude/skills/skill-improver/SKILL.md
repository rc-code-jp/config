# スキル改善ツール

エージェントスキル（SKILL.md）やエージェント定義ファイルを Anthropic 公式ガイドラインに基づいてレビューし、「発火条件つきの再現可能ワークフロー」として機能するよう改善するスキル。

## 対象ユースケース

このスキルは以下の3つのユースケースに特化する:

1. **新規スキルのレビュー**: 作成直後の SKILL.md が公式ガイドラインに沿っているか検証し、公開前の品質を担保する
2. **既存スキルの改善**: 運用中のスキルで発火不安定・品質のばらつき・ユーザー介入の多さがある場合に、原因を特定し改善する
3. **エージェント定義の最適化**: SKILL.md 形式ではないエージェント定義ファイル（`ai/agents/*.md` 等）の指示品質を評価し改善する

## 使う場面

- 「このスキルをレビューして」
- 「スキルの品質を改善して」
- 「SKILL.md を最適化して」
- 「エージェント定義を見直して」
- 「スキルのベストプラクティスに合わせて」

## 使用例

### 例 1: description が曖昧なスキルの改善

ユーザーの発話: 「このスキルをレビューして」

対象ファイルの状態:
```yaml
---
name: data-tool
description: データを処理するツール。
---
```

実行するアクション:
1. 構造チェック → name、デリミタは問題なし
2. description 品質評価 → What のみ。When（トリガーフレーズ）と Key capabilities が欠落
3. トリガリング評価 → 「データを処理」が曖昧でアンダートリガーのリスク大

レビュー結果:
```yaml
---
name: data-tool
description: CSV・JSON データの変換、集計、フィルタリングを行う。「CSV を集計して」「JSON を変換して」「データをフィルタして」と言われたときに使う。大規模データの統計分析やグラフ作成には使わない。
---
```

### 例 2: 運用安定性に問題があるスキルの改善

ユーザーの発話: 「このスキルが毎回うまく動かないので見直して」

対象ファイルの状態:
```yaml
---
name: deploy-helper
description: デプロイを手伝う。「デプロイして」と言われたときに使う。
---
# デプロイヘルパー
## 手順
1. デプロイ先を確認する
2. ビルドする
3. デプロイを実行する
4. 結果を報告する
```

実行するアクション:
1. description 品質評価 → What はあるが曖昧。Key capabilities がない。ネガティブトリガーもない
2. 運用安定性の評価:
   - 品質一貫性 → 各ステップの入出力依存が不明（「デプロイ先」はどこから取得？）
   - 自律進行性 → 分岐条件がない（staging と production で手順が違うはず）
   - 効率性 → 検証ゲートがない（ビルド失敗時の分岐が未定義）
3. 1責務検証 → 「ビルド」と「デプロイ」が混在。分割すべきか検討
4. テスト設計 → 完全に欠落

結果: 総合評価 C（重要度「高」2件: 入出力依存の断絶、分岐条件の欠落）。各ステップへの入出力追加、staging/production 分岐の明記、ビルド失敗時のエラーハンドリング、発火テスト10件の作成を提案

## 手順

### ステップ 1: 対象ファイルの特定と読み込み

入力: ユーザーの指示（ファイルパス指定 or 「全部見て」等）
出力: 対象ファイル一覧と各ファイルの内容、対象の種類（スキル or エージェント定義）

1. ユーザーが指定したファイル、またはプロジェクト内のスキル/エージェント定義を探す
2. 一般的な配置場所を確認:
   - `.claude/skills/*/SKILL.md`
   - `ai/skills/*/SKILL.md`
   - `ai/agents/*.md`
   - `.claude/agents/*.md`
3. 対象ファイルをすべて読み込む
4. **対象の種類を判定し、以降の評価パスを決定する**:

#### 分岐: 対象の種類による評価パスの切り替え

| 条件 | 評価パス | 適用するルール |
|---|---|---|
| `SKILL.md` という名前のファイル | **スキル評価パス** | フォルダ構造、YAML 全フィールド、段階的開示、references/ 分離を含む全項目を検証 |
| YAML フロントマター付きの `.md` ファイル（非 SKILL.md） | **エージェント定義評価パス** | フォルダ構造・references/ 分離は検証しない。name, description, tools フィールドと指示内容の品質を検証 |
| フロントマターなしの `.md` ファイル | **対象外** | スキル/エージェント定義ではないことをユーザーに報告し、確認を取る |

### ステップ 2: 構造チェック

入力: ステップ1の対象ファイルと種類判定結果
出力: 構造上の問題リスト（問題なしの場合は空）

まず `bash scripts/validate_frontmatter.sh <対象ファイルパス>` を実行し、形式的な検証を自動で行う。スクリプトが検出できない項目は以下を手動で検証する。詳細な良い例・悪い例は `references/best-practices.md` のセクション 0〜1 を参照。

#### ファイル命名・構造規則（手動検証）
- SKILL.md は正確な大文字小文字（case-sensitive）か（SKILL.MD、skill.md は不可）
- フォルダ名は kebab-case か（スペース、アンダースコア、大文字は NG）
- name フィールドがフォルダ名と一致しているか
- スキルフォルダ内に README.md が含まれていないか（ドキュメントは SKILL.md または references/ に置く）
- フォルダ構造が推奨パターンに沿っているか:
  - `SKILL.md`（必須）
  - `scripts/`（任意: 実行可能コード）
  - `references/`（任意: ドキュメント）
  - `assets/`（任意: テンプレート、フォント、アイコン等）

#### YAML フロントマター検証

##### 必須フィールド
- `---` デリミタが1行目と閉じ位置に正しく配置されているか
- `name`: kebab-case、スペースなし、大文字なし
- `description`: 1024文字以内。What（何をするか）+ When（いつ使うか）を必ず含む

##### 任意フィールドの検証（存在する場合）
- `license`: オープンソースの場合に指定（例: MIT, Apache-2.0）
- `allowed-tools`: ツールアクセスの制限（例: `"Bash(python:*) Bash(npm:*) WebFetch"`）
- `compatibility`: 環境要件の記述。1〜500文字以内。対象プロダクト、必要なシステムパッケージ、ネットワークアクセス要件など
- `metadata`: カスタムキーバリューペア。推奨フィールド:
  - `author`: 作成者名
  - `version`: バージョン番号
  - `mcp-server`: 連携する MCP サーバー名
  - `category`: カテゴリ
  - `tags`: タグ（リスト形式）
  - `documentation`: ドキュメント URL
  - `support`: サポート連絡先

##### セキュリティ制約
- XML タグ（`<` `>`）が含まれていないか（フロントマターはシステムプロンプトに展開されるため、インジェクションリスクがある）
- "claude" や "anthropic" がスキル名のプレフィックスに含まれていないか（予約語）
- YAML 内でコード実行を試みていないか（safe YAML parsing が適用される）
- 使用できる値の型: 文字列、数値、真偽値、リスト、オブジェクトのみ

#### description フィールド品質評価
以下の3要素がすべて含まれているか:
1. **何をするか**（What）: スキルの機能説明
2. **いつ使うか**（When）: トリガー条件・フレーズ
3. **主要な機能**（Key capabilities）: 差別化ポイント

悪い例（修正が必要）:
- 「プロジェクトを手伝います」→ 曖昧すぎる
- 「高度なドキュメントシステムを作成」→ トリガーフレーズがない
- 「Project エンティティモデルを実装」→ 技術的すぎ、ユーザー視点のトリガーがない

### ステップ 3: 指示内容の品質評価

入力: ステップ1のファイル内容
出力: 指示品質の問題リスト（具体性、運用安定性、責務範囲、ユースケース数）

指示の書き方の詳細原則は `references/best-practices.md` のセクション5を参照。

#### 具体性と実行可能性
- 指示が具体的で実行可能（actionable）か
- 曖昧な表現（「適切に処理する」「ちゃんと確認する」）がないか
- 各ステップが明確に定義されているか

改善前:
```
データを検証してから進める。
```

改善後:
```
`python scripts/validate.py --input {filename}` を実行してデータ形式を確認する。
検証失敗時の一般的な原因:
- 必須フィールドの欠落（CSV に追加する）
- 日付形式の不一致（YYYY-MM-DD を使用する）
```

#### 運用安定性の評価（再現可能な運用）
対象スキルが実運用で安定するかを、以下の4基準で評価する:
1. **発火安定性**: 似た依頼で安定して発火する設計になっているか。description に十分なトリガーフレーズがあるか
2. **品質一貫性**: 毎回同じ品質で進む設計か。ステップが状態機械的に記述され、各ステップの入出力依存が明示されているか
3. **自律進行性**: ユーザーが逐一誘導しなくてよいか。判断基準・分岐条件が明文化されているか
4. **効率性**: 無駄なツール呼び出しや失敗が減る設計か。検証ゲートが適切に配置されているか

#### 1スキル1責務の検証
対象スキルが複数の責務を混在させていないか確認する:
- description に複数の独立した機能が列挙されていないか
- 1つのスキルで扱うユースケースが多すぎないか（3つ以内が目安）
- スコープが広すぎて過剰発火のリスクがないか

分岐: 責務が多すぎる場合 → スキルの分割を提案する

#### ユースケース設計の評価
対象スキルが想定するユースケースを確認する:
- ユースケースが明示的に定義されているか
- 3つ以内に絞られているか
- 各ユースケースが具体的か（「何でもできる」は NG）

#### 段階的開示（Progressive Disclosure）の活用
- SKILL.md はコア指示に集中しているか
- 詳細ドキュメントは `references/` に分離されているか
- SKILL.md は 5,000 語以下を目安にしているか

#### エラーハンドリング
- よくあるエラーケースが記載されているか
- 各エラーに対する原因と解決策が明示されているか

#### 使用例
- 具体的なユースケース例が含まれているか
- ユーザーの発話例が含まれているか
- 期待される結果が明示されているか

### ステップ 4: パターン適合性の評価

入力: ステップ1のファイル内容、ステップ3の品質評価結果
出力: 該当パターン名、パターン適合度の問題リスト

スキルが以下のどのパターンに該当するか判定し、パターンに沿った構造になっているか確認する:

| パターン | 使用条件 | 重要な技法 |
|---|---|---|
| 逐次ワークフロー | 特定の順序で行うマルチステップ処理 | 明示的なステップ順序、依存関係、各段階の検証 |
| 複数 MCP 連携 | 複数サービスにまたがるワークフロー | フェーズ分離、MCP間のデータ受け渡し、一元的エラー処理 |
| 反復改善 | 反復で品質が向上するアウトプット | 品質基準の明示、改善ループ、検証スクリプト |
| 文脈依存選択 | 同じ結果でも文脈により手段が異なる | 明確な判断基準、フォールバック、選択理由の透明性 |
| ドメイン専門知識 | ツールアクセス以上の専門知識が必要 | ドメイン知識の組み込み、行動前のチェック、監査証跡 |

### ステップ 5: トリガリングの評価

入力: ステップ2の description 検証結果
出力: トリガリングリスク評価（アンダー/オーバー/問題なし）と対策案

トリガリング問題の詳細な対処法は `references/best-practices.md` のセクション4を参照。

- **アンダートリガー**: description が曖昧すぎて発動しない可能性
  - 対策: キーワード追加、技術用語の明記、トリガーフレーズの充実
- **オーバートリガー**: description が広すぎて無関係なクエリで発動する可能性
  - 対策: ネガティブトリガーの追加（「〜には使わない」）、スコープの限定

### ステップ 6: テスト設計の評価

入力: ステップ1のファイル内容、ステップ5のトリガリング評価結果
出力: テスト設計の充足度評価と不足しているテスト観点の一覧

対象スキルが以下の3つのテスト観点を持っているか確認する:

#### 発火テスト
- 発火すべき依頼例が記載されているか（最低5件推奨）
- 言い換え表現でも発火するよう考慮されているか
- 発火すべきでない依頼例が記載されているか（最低5件推奨）
- ネガティブトリガーが description に含まれているか

#### 機能テスト
- 正常系の期待出力が明示されているか
- ツール呼び出しの成功条件が定義されているか
- エラーハンドリングのテストケースがあるか
- エッジケースが考慮されているか

#### ベースライン比較
- スキル導入前後で比較可能な指標が定義されているか:
  - 往復回数（スキルあり vs なし）
  - トークン消費量
  - ツール呼び出し回数
  - API 失敗率
  - ユーザー介入の必要回数
- 「良い気がする」ではなく「前より何が減ったか」で評価できる設計か

分岐: テスト設計が完全に欠落 → 最低限の発火テスト（5件発火すべき + 5件発火すべきでない）の作成を提案する

### ステップ 7: 改善レポートの出力

入力: ステップ2〜6の全問題リスト
出力: 総合評価（A〜D）、問題一覧、改善後のファイル案

以下の形式でレポートを出力する:

```
## スキルレビュー結果: [スキル名]

### 総合評価: [A / B / C / D]

### 検出された問題

#### [重要度: 高/中/低] 問題タイトル
- 現状: ...
- 問題点: ...
- 改善案: ...

### 改善後のファイル案
(具体的な修正内容をコードブロックで提示)
```

#### 総合評価の判定基準

チェックリストの合否結果に基づき、以下の基準で判定する:

| 評価 | 条件 | 意味 |
|---|---|---|
| A | 重要度「高」の問題が 0 件、かつ「中」が 1 件以下 | 運用可能。軽微な改善のみ |
| B | 重要度「高」の問題が 0 件、かつ「中」が 2 件以上 | 運用可能だが改善推奨 |
| C | 重要度「高」の問題が 1〜2 件 | 改善しないと運用で問題が出る |
| D | 重要度「高」の問題が 3 件以上、または description が機能していない | 根本的な再設計が必要 |

#### 問題の重要度判定基準

| 重要度 | 基準 |
|---|---|
| 高 | 発火しない、または発火後にワークフローが破綻する問題。description 欠陥、入出力依存の断絶、分岐条件の欠落、エラーハンドリング不在 |
| 中 | 運用はできるが品質や効率が低下する問題。ユースケース未定義、テスト設計不足、段階的開示の不活用、1責務違反 |
| 低 | 品質向上の余地がある問題。命名慣習、見出し言語、参照リンク不足 |

### ステップ 8: 改善の実施

入力: ステップ7のレポートとユーザーの承認
出力: 編集済みファイル

ユーザーの承認を得たら、以下を実行する:
1. 改善案に基づきファイルを編集する
2. 編集後のファイルを表示して確認を促す
3. 追加の改善が必要か確認する

## トラブルシューティング

### 対象ファイルが見つからない
原因: スキルやエージェント定義が標準的なパスに配置されていない。
解決策:
1. ユーザーにファイルパスを確認する
2. プロジェクトルートから `SKILL.md` や `AGENTS.md` を再帰的に検索する
3. `.md` 拡張子のフロントマター付きファイルをすべて候補として提示する

### YAML フロントマターのパースに失敗
原因: `---` デリミタの欠落、クォートの閉じ忘れ、インデントの不整合。
解決策:
1. ファイル先頭の `---` が1行目にあるか確認する
2. 閉じ `---` が存在するか確認する
3. description に特殊文字（`:` `#` `"` `'`）が含まれる場合はクォートで囲む
4. 修正案をユーザーに提示してから適用する

### レビュー対象が大量にある場合
原因: プロジェクト内に多数のスキル/エージェントファイルが存在する。
解決策:
1. まず一覧を提示し、ユーザーにレビュー対象を選択してもらう
2. 優先度の提案: description が曖昧なもの、構造に問題があるものを優先
3. 一度にすべてをレビューせず、1〜3ファイルずつ進める

## 自身のテスト設計

このスキル自身の発火テスト（5件発火すべき + 5件発火すべきでない）、機能テスト（正常系3件 + エラー系2件）、ベースライン比較は `references/self-test-cases.md` を参照。

## 評価基準チェックリスト

レビュー時に以下をすべて確認する:

### 構造とファイル
- [ ] SKILL.md の命名が正確（case-sensitive）
- [ ] フォルダ名が kebab-case
- [ ] フォルダ内に README.md が含まれていない
- [ ] フォルダ構造が推奨パターン（scripts/, references/, assets/）に沿っている

### YAML フロントマター
- [ ] `---` デリミタが正しく配置
- [ ] name が kebab-case でフォルダ名と一致
- [ ] description が What + When + Capabilities を含む
- [ ] description が 1024 文字以内
- [ ] XML タグ（`<` `>`）不使用
- [ ] "claude" / "anthropic" プレフィックス不使用
- [ ] YAML 内でコード実行を試みていない
- [ ] 任意フィールド（license, allowed-tools, compatibility, metadata）が正しい形式
- [ ] compatibility が 500 文字以内（使用時）

### 内容
