# skill-improver 自身のテスト設計

## 発火テスト

### 発火すべき依頼（5件）

1. 「このスキルをレビューして」
2. 「SKILL.md を見直してほしい」
3. 「エージェント定義を最適化して」
4. 「このスキルの運用が安定しないので改善して」
5. 「スキルのベストプラクティスに合っているか確認して」

### 発火すべきでない依頼（5件）

1. 「このプルリクエストをレビューして」→ コードレビュー／PR レビュー
2. 「この関数のコードを見直して」→ コードレビュー
3. 「README.md を改善して」→ 一般的なドキュメントレビュー
4. 「新しいスキルを1から作って」→ スキル作成（レビューではない）
5. 「Python のテストを書いて」→ コード作成

## 機能テスト

### 正常系

#### ケース 1: description が曖昧なスキル
入力: `description: データを処理するツール。` のみの SKILL.md
期待出力:
- 総合評価 D（description が機能していない）
- 重要度「高」: description に When と Capabilities が欠落
- 改善後の description 案が What + When + Capabilities + ネガティブトリガーを含む

#### ケース 2: 構造は良いが運用安定性に問題があるスキル
入力: description は適切だがステップに入出力依存・分岐条件がないスキル
期待出力:
- 総合評価 C（重要度「高」1〜2件）
- 重要度「高」: 入出力依存の断絶、分岐条件の欠落
- 各ステップへの入出力追加案

#### ケース 3: 高品質なスキル
入力: 全チェック項目を満たす SKILL.md
期待出力:
- 総合評価 A
- 重要度「高」0件、「中」0〜1件
- 軽微な改善提案のみ

### エラー系

#### ケース 4: YAML フロントマターが壊れている
入力: `---` デリミタが欠落したファイル
期待出力:
- トラブルシューティングの「YAML フロントマターのパースに失敗」に沿った対応
- デリミタ修正案の提示

#### ケース 5: 対象ファイルが見つからない
入力: 「スキルをレビューして」だがプロジェクト内にスキルファイルがない
期待出力:
- トラブルシューティングの「対象ファイルが見つからない」に沿った対応
- ファイルパスの確認をユーザーに求める

## ベースライン比較

このスキルの導入効果を以下の指標で測定する:

| 指標 | スキルなし（手動レビュー） | スキルあり |
|---|---|---|
| レビュー観点の網羅性 | レビュアー依存でばらつく | チェックリスト 30 項目を毎回全件確認 |
| レビュー所要時間（往復メッセージ数） | 5〜10 往復 | 1〜2 往復 |
| 改善提案の具体性 | 「もっと具体的にしたほうがいい」等の曖昧なフィードバック | 改善前/後のコード例付き |
| 評価の再現性 | レビュアーごとに異なる | A〜D の定量基準で一貫 |
