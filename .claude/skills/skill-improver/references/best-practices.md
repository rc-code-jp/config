# スキル構築ベストプラクティス詳細リファレンス

Anthropic 公式ガイド「The Complete Guide to Building Skills for Claude」に基づく。

## 0. YAML フロントマター全フィールドリファレンス

### 必須フィールド

```yaml
---
name: skill-name-in-kebab-case
description: 何をするか、いつ使うか。具体的なトリガーフレーズを含める。
---
```

### 全任意フィールドの完全な例

```yaml
---
name: skill-name
description: [必須の説明文]
license: MIT                                          # 任意: オープンソースライセンス（MIT, Apache-2.0 等）
allowed-tools: "Bash(python:*) Bash(npm:*) WebFetch"  # 任意: ツールアクセス制限
compatibility: Claude.ai and Claude Code. Requires Python 3.10+. # 任意: 環境要件（1-500文字）
metadata:                                             # 任意: カスタムフィールド
  author: Company Name
  version: 1.0.0
  mcp-server: server-name
  category: productivity
  tags: [project-management, automation]
  documentation: https://example.com/docs
  support: support@example.com
---
```

### フィールド詳細

| フィールド | 必須 | 制約 | 説明 |
|---|---|---|---|
| `name` | はい | kebab-case、スペース・大文字不可、フォルダ名と一致 | スキルの識別名 |
| `description` | はい | 1024文字以内、XML タグ禁止、What + When を含む | スキルの説明とトリガー条件 |
| `license` | いいえ | 標準的なライセンス識別子 | オープンソース配布時に使用 |
| `allowed-tools` | いいえ | ツール名のスペース区切り | スキルがアクセスできるツールを制限 |
| `compatibility` | いいえ | 1〜500文字 | 対象プロダクト、必要パッケージ、ネットワーク要件 |
| `metadata` | いいえ | 任意のキーバリューペア | author, version, mcp-server 等の補足情報 |

### セキュリティ制約

- XML アングルブラケット（`<` `>`）禁止: フロントマターはシステムプロンプトに展開されるため、悪意あるコンテンツがインジェクションされるリスクがある
- `claude` や `anthropic` をスキル名のプレフィックスに使用禁止（予約語）
- YAML 内でのコード実行禁止（safe YAML parsing が適用される）
- 許可される YAML 値の型: 文字列、数値、真偽値、リスト、オブジェクトのみ

### ファイル構造ルール

```
your-skill-name/
├── SKILL.md          # 必須 - メインスキルファイル（case-sensitive）
├── scripts/          # 任意 - 実行可能コード
│   ├── process_data.py
│   └── validate.sh
├── references/       # 任意 - ドキュメント
│   ├── api-guide.md
│   └── examples/
└── assets/           # 任意 - テンプレート、フォント、アイコン等
    └── report-template.md
```

- README.md をスキルフォルダ内に含めない（ドキュメントは SKILL.md または references/ に置く）
- GitHub で配布する場合は、リポジトリレベルの README はスキルフォルダの外に配置する

## 1. Description フィールドの書き方

### 構造

`[何をするか] + [いつ使うか] + [主要機能]`

### 良い例

```yaml
description: Figma デザインファイルを分析し、開発者向けハンドオフドキュメントを生成する。.fig ファイルのアップロード時、「デザインスペック」「コンポーネントドキュメント」「デザインからコードへのハンドオフ」を求められた時に使用する。
```

```yaml
description: Linear プロジェクトワークフローを管理する。スプリント計画、タスク作成、ステータス追跡を含む。「スプリント」「Linear タスク」「プロジェクト計画」「チケット作成」と言われた時に使用する。
```

### 悪い例と改善方法

| 悪い例 | 問題 | 改善案 |
|---|---|---|
| `プロジェクトを手伝います` | 曖昧すぎる | 具体的な機能とトリガーを記載 |
| `高度なドキュメントシステムを作成` | トリガー不在 | ユーザーの発話例を追加 |
| `Project エンティティモデルを実装` | 技術的すぎ | ユーザー視点の表現に変更 |

## 2. ユースケースカテゴリ

### カテゴリ 1: ドキュメント・アセット作成
- 一貫した高品質なアウトプット生成
- スタイルガイドやブランド基準の組み込み
- テンプレート構造による一貫性
- 最終化前の品質チェックリスト

### カテゴリ 2: ワークフロー自動化
- 一貫した方法論のマルチステップ処理
- バリデーションゲート付きのステップ
- テンプレートによる共通構造
- 反復的な改善ループ

### カテゴリ 3: MCP 強化
- MCP サーバーの複数呼び出しの連携
- ドメイン専門知識の組み込み
- ユーザーが指定する必要があるコンテキストの提供
- 一般的な MCP 問題のエラーハンドリング

## 3. 成功基準

### 定量的指標
- スキルが関連クエリの 90% でトリガーされる
- ワークフローが X 回のツール呼び出しで完了する
- ワークフローあたり API 呼び出し失敗 0 件

### 定性的指標
- ユーザーが次のステップについて Claude にプロンプトする必要がない
- ワークフローがユーザー修正なしで完了する
- セッション間で一貫した結果

## 4. トリガリング問題の対処

### アンダートリガー（発動しない）
信号: スキルが自動的にロードされない、ユーザーが手動で有効化
対策:
- description にキーワードを追加（特に技術用語）
- より多くのトリガーフレーズを含める
- ユーザーが実際に言う表現を使う

### オーバートリガー（過剰に発動）
信号: 無関係なクエリでスキルがロードされる
対策:
1. ネガティブトリガーを追加
```yaml
description: CSV ファイルの高度なデータ分析。統計モデリング、回帰、クラスタリング用。単純なデータ探索には使わない（data-viz スキルを使う）。
```
2. より具体的にする
3. スコープを明確にする

## 5. 指示の書き方

### 重要な原則
- 重要な指示はファイルの上部に配置する
- `## Important` や `## Critical` ヘッダーを使う
- 必要に応じて重要ポイントを繰り返す
- 曖昧な言葉を避け、具体的に記述する

### 検証は言語でなくコードで
重要な検証にはスクリプトをバンドルして使う。コードは確定的、言語の解釈は不確定的。

### モデルの「怠惰」への対処
明示的な励ましを追加:
```
## パフォーマンスノート
- 徹底的に実行すること
- 品質はスピードより重要
- 検証ステップをスキップしない
```
※ これは SKILL.md よりもユーザープロンプトに追加する方が効果的。

## 6. 運用安定性の設計原則

### スキルの価値は「再現可能な運用」
良いスキルは、毎回うまくいった会話を再現するためのパッケージ。4つの基準で評価する:

1. **発火安定性**: 似た依頼で安定して発火する
2. **品質一貫性**: 毎回同じ品質で進む
3. **自律進行性**: ユーザーが逐一誘導しなくてよい
4. **効率性**: 無駄なツール呼び出しや失敗が減る

### 1スキル1責務
- 1つのスキルに複数の独立した機能を詰め込まない
- 広いスキルほど description が曖昧になり、過剰発火しやすい
- ユースケースは 3 つ以内に絞る

### 逐次ワークフロー（状態機械的記述）
うまいスキルは状態機械的に書かれている:
- 各ステップに入力と出力を明記
- ステップ間の依存関係を明示
- 分岐条件を具体的に定義（例: ファイルサイズ > 10MB → クラウド保存）

### 検証のスクリプト化
- 「ちゃんと確認して」は曖昧。コードに落とす方が決定的
- 曖昧な品質基準ほど `scripts/` にスクリプトとしてバンドルする
- コードは確定的、言語の解釈は不確定的

## 7. テスト設計

### 発火テスト
目的: スキルが適切なタイミングで発火するか確認

テストケース構成:
- 発火すべき依頼: 5 件以上（直接的な表現 + 言い換え表現）
- 発火すべきでない依頼: 5 件以上（無関係な話題 + 類似だがスコープ外の話題）

例:
```
発火すべき:
- "ProjectHub のワークスペースを作って"
- "新しいプロジェクトを ProjectHub に立ち上げたい"
- "Q4 計画用の ProjectHub プロジェクトを準備して"

発火すべきでない:
- "サンフランシスコの天気は？"
- "Python のコードを書いて"
- "スプレッドシートを作って"（ProjectHub がシート機能を持たない場合）
```

### 機能テスト
目的: 発火後に正しく動作するか確認

テストケース構成:
- 正常系: 有効な入力で期待通りの出力が得られるか
- 異常系: API 失敗、認証切れ、入力不足でエラーハンドリングが動くか
- エッジケース: 境界値、空入力、大量データでの挙動

### ベースライン比較
目的: スキル導入で本当に改善されたか定量的に評価

比較指標:
| 指標 | スキルなし | スキルあり |
|---|---|---|
| 往復メッセージ数 | 15 | 2 |
| トークン消費量 | 12,000 | 6,000 |
| API 失敗回数 | 3 | 0 |
| ユーザー介入回数 | 多数 | 0 |

「良い気がする」ではなく「前より何が減ったか」で評価する。

## 8. コンテキストサイズの最適化

- SKILL.md は 5,000 語以下に保つ
- 詳細ドキュメントは `references/` に分離
- 同時有効スキル数は 20-50 以下を推奨
- Progressive Disclosure の3レベルを活用:
  1. YAML フロントマター（常にロード）
  2. SKILL.md 本文（関連時にロード）
  3. リンクファイル（必要時にのみアクセス）
